## 二、技术栈决策

### 2.1 数据库双数据源方案

**结论：MySQL（业务数据）+ PostgreSQL（向量数据），双数据源共存**

- `MySQL`：承载所有业务数据（用户、简历、面试会话、知识库元数据等）
- `PostgreSQL + pgvector`：仅承载向量嵌入数据（`vector_store` 表）

**理由：**
- interview-guide 的向量能力完全依赖 Spring AI 的 `PgVectorStore`，切换至 MySQL 需要自研向量检索
- MySQL 8.0 官方尚无稳定的向量索引支持（`VECTOR` 类型在 9.x 才有），生产可靠性不足
- pgvector 已是向量存储的事实标准，Spring AI 内置支持，迁移成本极低
- 双数据源在 Spring Boot 中是成熟方案，配置清晰，边界明确

**数据源划分：**

```
MySQL (主数据源，MyBatis-Plus)
├── 简历相关表：t_resume, t_resume_analysis
├── 面试相关表：t_interview_session, t_interview_answer
├── 知识库元数据表：t_knowledge_base, t_rag_chat_session, t_rag_chat_message
├── LLM Provider 表：t_llm_provider, t_llm_global_setting
├── 面试日程表：t_interview_schedule
└── 语音面试表：t_voice_session, t_voice_message, t_voice_evaluation

PostgreSQL (向量数据源，Spring AI PgVectorStore)
└── vector_store 表（Spring AI 自动管理，HNSW 索引，1024 维 COSINE）
```

### 2.2 Redis Stream vs RocketMQ 评估

**结论：暂定 Redis Stream**

#### 两者对比

| 维度 | Redis Stream | RocketMQ |
|------|-------------|---------|
| **部署成本** | 复用现有 Redis | 需要额外部署 NameServer + Broker |
| **消息可靠性** | 支持 ACK，消息持久化（appendonly） | 更强，支持事务消息、精确一次 |
| **延时消息** | 不支持 | 支持（定时/延时级别） |
| **消息顺序** | 单分区有序 | 支持严格顺序消费 |
| **消息过滤** | 需手动实现 | 支持 Tag/SQL 过滤 |
| **监控运维** | 基础，需自行观测 | 完善的 Dashboard（RocketMQ Console） |
| **Spring Boot 集成** | 通过 Redisson 或手写 | 官方 rocketmq-spring-boot-starter |
| **学习曲线** | 低 | 中等 |
| **适用场景** | 中低流量、简单任务队列 | 高流量、高可靠、复杂消息场景 |

#### 面试平台的异步任务特征

本项目涉及 4 条异步管道：

| 管道 | 特征 | Redis Stream 是否足够 |
|------|------|----------------------|
| 知识库向量化 | 文档解析 + 嵌入 API 调用，中等频率 | ✅ 足够 |
| 简历分析 | AI 评分，低频 | ✅ 足够 |
| 面试评估 | AI 评估，中等频率 | ✅ 足够 |
| 语音面试评估 | AI 多轮评估，低频 | ✅ 足够 |

**决策依据：**
1. 面试平台用户量有限，消息吞吐不是瓶颈
2. 不需要延时消息、顺序消息等高级特性
3. 复用 Redis 减少基础设施依赖，部署更简单
4. mhd-interview 项目已有 RocketMQ 依赖，若后期需要可平滑迁移

> **后期升级信号**：当日活用户数超 5000，或需要消息延时/事务/复杂路由时，再迁移至 RocketMQ。

### 2.3 ORM 框架：JPA → MyBatis-Plus

interview-guide 使用 Spring Data JPA，mhd-interview 使用 MyBatis-Plus。
迁移时 Entity 改为 MyBatis-Plus 的 `@TableName`/`@TableId` 注解体系，Repository 改为 `BaseMapper`。

**核心差异速查：**

| 概念 | JPA（源） | MyBatis-Plus（目标） |
|------|-----------|---------------------|
| 实体注解 | `@Entity`, `@Table`, `@Id` | `@TableName`, `@TableId`, `@TableField` |
| 基础接口 | `JpaRepository<T, ID>` | `BaseMapper<T>` / `IService<T>` |
| 自定义查询 | `@Query("SELECT ...")` | `@Select(...)` 或 XML Mapper |
| 分页 | `PageRequest` / `Page<T>` | `Page<T>` + `selectPage()` |
| DDL 自动创建 | `spring.jpa.ddl-auto=update` | 不支持，需手写 SQL 或 Liquibase |
| 条件构造 | `Specification<T>` | `QueryWrapper<T>` / `LambdaQueryWrapper<T>` |

> **注意**：MyBatis-Plus 不会自动建表，迁移时需要手写建表 SQL。

### 2.4 Java 版本：17 → 21（强烈建议升级）

**结论：建议升级 Java 版本至 21**

#### 虚拟线程 vs 线程池差异分析

| 维度 | Java 17 线程池 | Java 21 虚拟线程 |
|------|---------------|----------------|
| **内存占用** | 每线程约 512KB～1MB | 每虚拟线程约 2～8KB |
| **最大并发** | 通常几百到几千 | 可轻松达到百万 |
| **配置复杂度** | 需手动配置核心数、最大数、队列 | 无需配置，JVM 自动调度 |
| **阻塞 IO 行为** | 阻塞时占用平台线程 | 阻塞时自动让出平台线程（mount/unmount） |
| **适用场景** | CPU 密集型 | IO 密集型（**本项目的典型场景**） |

#### 对 AI 面试平台的具体影响

本项目 IO 密集场景极多：
- **LLM API 调用**：每次请求等待 3～30 秒，线程持续阻塞
- **向量化 API 调用**：嵌入模型调用网络 IO
- **文件解析**：磁盘 + 网络 IO
- **WebSocket 语音面试**：大量并发连接

**使用线程池的痛点：**
```
假设 LLM 调用平均 10 秒，线程池 200 个线程：
最大并发只有 200 个请求，多余请求排队或拒绝
每个线程占 1MB 内存 → 200 线程消耗 200MB

使用虚拟线程：
同样 200 个物理线程，可支持 >10000 个并发请求
虚拟线程在等待 LLM 响应时自动让出平台线程
```

**升级成本极低：**
```xml
<!-- pom.xml 只需改一行 -->
<java.version>21</java.version>

<!-- Spring Boot 配置开启虚拟线程 -->
spring:
  threads:
    virtual:
      enabled: true
```

Spring Boot 3.5.9 完整支持 Java 21 和虚拟线程，升级风险极小。
Java 21 是 LTS 版本，生产已非常稳定。

> **最终建议：升级至 Java 21，开启虚拟线程。** 这对 LLM 调用、向量化等 IO 密集型场景有显著性能提升，且改动成本接近于零。

### 2.5 Spring AI 版本适配

| 项 | interview-guide | mhd-interview |
|----|----------------|--------------|
| Spring Boot | 4.0 | 3.5.9 |
| Spring AI | 2.0 | 1.0.x |

Spring AI 1.0.x 与 2.0 核心 API 基本兼容，主要差异：
- BOM 坐标：`spring-ai-bom:1.0.0`（对应 SB 3.x）
- PgVector 依赖：`spring-ai-pgvector-store-spring-boot-autoconfigure`
- ChatClient API 在 1.0.x 中已存在，迁移改动小

## 四、模块迁移计划

### 4.1 各模块迁移详情

#### 4.1.1 LLM Provider 管理模块（P0，最先迁移）

**功能说明：** 其他所有 AI 功能模块的前提，必须最先完成。

**迁移内容：**

| 源类（JPA） | 目标类（MyBatis-Plus） | 改动说明 |
|------------|----------------------|---------|
| `LlmProviderEntity`（@Entity） | `LlmProviderEntity`（@TableName） | JPA 注解 → MP 注解 |
| `LlmGlobalSettingEntity` | `LlmGlobalSettingEntity` | 同上 |
| `LlmProviderRepository extends JpaRepository` | `LlmProviderMapper extends BaseMapper` | 接口改变 |
| `LlmGlobalSettingRepository` | `LlmGlobalSettingMapper` | 同上 |
| `LlmProviderConfigService`（读写 yml 文件） | `LlmProviderConfigService`（读写数据库） | **优化**：配置持久化改为 MySQL，不再依赖本地文件 |

### 5.2 Redis Stream 异步任务框架（对应 interview-common-ai 模块）

```java
// AbstractStreamProducer.java
public abstract class AbstractStreamProducer<T> {
    
    @Autowired
    protected RedissonClient redissonClient;
    
    protected void sendTask(T payload) {
        RStream<String, String> stream = redissonClient.getStream(streamKey());
        Map<String, String> message = buildMessage(payload);
        stream.addAsync(StreamAddArgs.entries(message)
                .maxLen(AsyncTaskStreamConstants.STREAM_MAX_LEN));
        log.info("Task sent to stream: {}, task: {}", streamKey(), taskDisplayName());
    }
    
    protected abstract String streamKey();
    protected abstract String taskDisplayName();
    protected abstract Map<String, String> buildMessage(T payload);
    protected void onSendFailed(T payload, Exception e) {
        log.error("Failed to send task to stream: {}", streamKey(), e);
    }
}

// AbstractStreamConsumer.java
public abstract class AbstractStreamConsumer<T> {
    
    @PostConstruct
    public void init() {
        // 创建 Consumer Group
        createGroupIfAbsent();
        // 启动消费线程（虚拟线程或线程池）
        Thread.ofVirtual().name("stream-consumer-" + consumerPrefix())
              .start(this::consumeLoop);
    }
    
    private void consumeLoop() {
        while (!Thread.currentThread().isInterrupted()) {
            // 批量读取消息
            Map<StreamMessageId, Map<String, String>> messages = readMessages();
            for (Map.Entry<StreamMessageId, Map<String, String>> entry : messages.entrySet()) {
                handleWithRetry(entry.getKey(), entry.getValue());
            }
        }
    }
    
    protected abstract void processMessage(T payload, StreamMessageId msgId);
    protected abstract String streamKey();
    protected abstract String groupName();
    protected abstract String consumerPrefix();
    protected abstract T deserialize(Map<String, String> message);
}
```

## 附录：迁移注意事项

### A. 已有 interview-common-web 中的 GlobalExceptionHandler 需扩展

当前 mhd-interview 的 `GlobalExceptionHandler` 需要新增以下异常处理：

```java
// 新增：限流异常
@ExceptionHandler(RateLimitExceededException.class)
public BaseResponse<?> handleRateLimit(RateLimitExceededException e) {
    return BaseResultUtils.fail(8001, "请求过于频繁，请稍后再试");
}

// 新增：AI 服务超时
@ExceptionHandler(TimeoutException.class)
public BaseResponse<?> handleTimeout(TimeoutException e) {
    return BaseResultUtils.fail(7002, "AI 服务超时，请稍后重试");
}
```

### B. 扫描路径调整

`InterviewApplication.java` 的 `@SpringBootApplication` 包扫描范围，需要覆盖 `interview-common-ai` 中的 AOP 组件：

```java
@SpringBootApplication(scanBasePackages = {
    "com.mhd.interview",
    "com.mhd.interview.common"     // 扫描 libs 中的 common-ai 模块
})
@EnableScheduling
public class InterviewApplication { ... }
```

### C. 关于 MapStruct 的决策

interview-guide 使用 MapStruct 做 Entity↔DTO 映射。mhd-interview 目前没有 MapStruct，有两种选择：
1. **引入 MapStruct**：映射代码更简洁，推荐用于字段多的 Entity
2. **手写 Convert 类**：更直观，对于字段少的简单 Entity 足够

**建议：** 引入 MapStruct（已在 § 5.4 中包含依赖），在 `convert/` 包中放置 `@Mapper` 接口。

### D. 关于 Prompt 注入安全

interview-guide 中有 `PromptSanitizer` 类做 Prompt 注入防护，迁移时需一并引入到 `interview-common-ai`，在所有用户输入传入 LLM 前都需要调用清洗。

### E. 日志规范对齐

interview-guide 使用结构化日志（SLF4J），mhd-interview 也是，规范一致，直接沿用：

```java
// 结构化：字段名=值
log.info("Session created: sessionId={}, provider={}", id, provider);

// 异常日志：异常作为最后参数
log.error("Evaluation failed: sessionId={}", id, e);

// 禁止：
// log.error("Error: {}", e.getMessage())  // 丢失堆栈
```
